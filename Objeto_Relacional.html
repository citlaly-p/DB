I · Fundamentos

Introducción

El modelo Objeto-Relacional (OR) combina la robustez del modelo relacional con las capacidades de modelado de la orientación a objetos.

Enfoque Híbrido
Representa una evolución del estándar SQL (SQL:1999 y posteriores) para manejar datos complejos sin perder la integridad transaccional del mundo relacional.
Ventajas OR Limitaciones OR
Mayor flexibilidad y modelado más natural. Es un enfoque más práctico que teórico.
Robustez y confiabilidad heredadas del relacional. La compatibilidad con sistemas limita la integración.
Combina SQL con tipos, herencia y métodos. Diseño de esquemas OO puede ser complicado.
II · Estructuras

Modelo de Datos: Tipos Estructurados

Permiten definir atributos complejos que rompen con la atomicidad del modelo relacional tradicional.

1. Tipo Fila (Row type)

Representa una tupla compleja definida por el usuario que puede usarse como tipo de dato en una columna.

CREATE TYPE Publisher AS ROW (
  name varchar(20),
  branch varchar(20)
);

2. Tipo Arreglo (Array type)

Una colección de valores del mismo tipo con un tamaño máximo definido.

authors varchar(20) ARRAY[10]; -- Colección de hasta 10 autores

3. Relación anidada

Permite que una tabla contenga otra tabla como atributo, permitiendo jerarquías de datos complejas.

CREATE TYPE AddressTypeTable AS TABLE OF AddressType;

CREATE TYPE Person AS (
  name CHAR(30),
  address AddressTypeTable
); --
III · Identidad

Identidad y Referencias

Las referencias permiten que una tupla apunte a otra celda en lugar de incluir físicamente los datos, funcionando como punteros gestionados por la base de datos.

Tipos de OID (Identificadores de Objeto)
  • System generated: El sistema lo genera de forma única y automática.
  • User generated: El usuario provee y gestiona el valor manualmente.
  • Derived: Se genera automáticamente derivándolo de la llave primaria.

Ejemplos de creación con OID

-- OID generado por el sistema
CREATE TABLE people OF Person REF IS oid SYSTEM GENERATED;

-- OID derivado de la llave primaria
CREATE TABLE people OF Person REF IS oid DERIVED;
IV · Comportamiento

Métodos

Operaciones asociadas a un tipo que constan de una firma (declaración en el tipo) y un cuerpo (definición por separado).

⚙️

Constructores

Pueden ser por defecto (automáticos) o definidos por el usuario.

🧠

Lógica

Los métodos generales operan sobre la instancia actual usando self.

-- 1. Declaración de la firma
CREATE TYPE Employee AS (name varchar(20), salary integer)
METHOD giveraise (percent integer);

-- 2. Definición del cuerpo
CREATE METHOD giveraise (percent integer) FOR Employee
BEGIN
  SET self.salary = self.salary + (self.salary * percent) / 100;
END;
V · Jerarquías

Herencia de Tipos y Tablas

Herencia de tipos (UNDER)

Los subtipos heredan todos los atributos y pueden redefinir métodos del supertipo original.

CREATE TYPE Student UNDER Person (
  degree varchar(20),
  department varchar(20)
);

Herencia de tablas (INHERITS)

Las subtablas heredan columnas de la supertabla. Una tupla insertada en la subtabla es visible automáticamente en la supertabla.

CREATE TABLE students (
  degree varchar(20)
) INHERITS (people); -- Herencia de estructura de tabla
VI · Operaciones

Manipulación de Datos con SQL-OR

La manipulación de datos en SQL-OR extiende el estándar tradicional para permitir la gestión de tipos complejos y colecciones de forma nativa.

1. Inserción

Al insertar datos, se deben utilizar constructores específicos para tipos estructurados (ROW) y colecciones (ARRAY).

INSERT INTO Books VALUES (
  'Compilers',
  ARRAY['Smith','Jones'], -- Inserción de colección
  ROW('McGraw Hill','New York') -- Inserción de tupla (Row Type)
);

2. Consultas y Navegación

El acceso a la información dentro de objetos requiere operadores especializados que permitan profundizar en la estructura de los datos.

🎯

Notación Punto

Acceso a campos internos de un tipo estructurado. Ej: (publisher).name.

🔗

Dereferencing (->)

Operador para seguir una referencia (OID) y acceder a los datos del objeto apuntado.

📦

Unnesting

Expande una colección a filas individuales mediante la función UNNEST.

Ejemplo de Unnesting SELECT B.title, A FROM books AS B, UNNEST(B.authors) AS A;
