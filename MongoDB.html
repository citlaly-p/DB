I · Introducción

¿Qué es MongoDB? Introducción y contexto

MongoDB es un sistema gestor de bases de datos que pertenece a la categoría de las bases de datos NoSQL (del inglés "Not Only SQL"), y más específicamente, al modelo de bases de datos orientadas a documentos. A diferencia de las bases de datos relacionales tradicionales (como MySQL o PostgreSQL), MongoDB no almacena la información en tablas con filas y columnas, sino en documentos — unidades de información más flexibles y expresivas.

🧩 Analogía
Imagina que en una base de datos relacional los datos viven en hojas de cálculo: todo en columnas fijas, perfectamente alineado. MongoDB, en cambio, se parece más a una carpeta llena de fichas de papel: cada ficha puede tener los campos que necesite, en el orden que quiera, y pueden ser diferentes entre sí.

¿Qué es BSON?

MongoDB utiliza internamente un formato llamado BSON (Binary JSON), que es una extensión del popular formato JSON (JavaScript Object Notation). Para comprender BSON, primero hay que entender JSON:

JSON es una forma de representar datos usando texto legible para humanos, organizando la información en pares de clave: valor, encerrados entre llaves { }. Es el mismo formato que usan muchas aplicaciones web para transmitir datos.

BSON agrega sobre JSON la capacidad de almacenar tipos de datos adicionales (como fechas y datos binarios) y hace que el almacenamiento y la búsqueda sean más eficientes al ser un formato binario (legible eficientemente por computadoras, aunque no a simple vista por humanos).

📌 Concepto Clave
JSON = formato de texto legible para humanos, con pares clave-valor. BSON = versión binaria y enriquecida de JSON, usada internamente por MongoDB para mayor velocidad y soporte de más tipos de datos.

MongoDB es popular especialmente en el desarrollo de aplicaciones modernas por dos razones fundamentales: su flexibilidad (no exige definir la estructura de los datos con anticipación) y su escalabilidad (puede manejar enormes cantidades de datos distribuyéndolos en múltiples servidores).

II · Ventajas

Ventajas principales de MongoDB

MongoDB ha ganado gran popularidad en la industria del software por cuatro ventajas fundamentales:

1. Esquema flexible

En una base de datos relacional, antes de guardar cualquier dato hay que definir exactamente qué campos tendrá cada registro y de qué tipo serán. En MongoDB, no es obligatorio. Dos documentos dentro de la misma colección pueden tener campos completamente diferentes sin que eso cause ningún error. Esta característica es enormemente útil cuando los datos evolucionan con el tiempo o cuando provienen de fuentes variadas.

2. Consultas rápidas

MongoDB utiliza índices (estructuras de búsqueda acelerada, similares al índice al final de un libro) que permiten encontrar documentos de forma muy eficiente, incluso en colecciones con millones de registros. Además, al almacenar datos relacionados juntos dentro de un mismo documento, evita la necesidad de costosas operaciones de unión entre tablas (joins) que son comunes en bases de datos relacionales.

3. Alta disponibilidad con replicación

MongoDB puede configurarse para mantener copias automáticas de los datos en múltiples servidores, un mecanismo llamado replicación. Si un servidor falla, otro toma el control automáticamente sin interrumpir el servicio. Esto es fundamental para aplicaciones críticas que no pueden permitirse tiempo de inactividad.

🧩 Analogía
La replicación es como tener varias fotocopias del mismo expediente en distintos archiveros. Si uno se rompe o se pierde, siempre hay una copia disponible.

4. Escalabilidad horizontal

Cuando una base de datos crece demasiado para un solo servidor, existen dos estrategias: la escalabilidad vertical (comprar un servidor más potente y caro) y la escalabilidad horizontal (agregar más servidores baratos y distribuir los datos entre ellos). MongoDB está diseñado desde sus cimientos para la segunda estrategia, a través de un mecanismo llamado sharding (fragmentación de datos). Esto lo hace ideal para aplicaciones que manejan millones de usuarios o terabytes de información.

III · Casos de uso

Casos de uso

MongoDB no es la solución adecuada para todo tipo de problema. Brilla especialmente en los siguientes escenarios:

  • Aplicaciones web con datos semi-estructurados: cuando cada usuario o entidad puede tener atributos distintos. Por ejemplo, un perfil de usuario en una red social donde unos tienen foto de perfil, otros tienen dirección, y otros tienen ambos o ninguno.
  • Almacenamiento de catálogos o registros de eventos: en un catálogo de productos de e-commerce, una televisión puede tener "resolución" y "tamaño de pantalla", mientras que un libro tiene "autor" y "ISBN". Con MongoDB, ambos conviven en la misma colección sin forzar campos vacíos.
  • Aplicaciones de IoT (Internet of Things): los sensores emiten datos constantemente y en formatos variados. MongoDB puede ingerir estos flujos de datos de forma eficiente.
  • E-commerce: gestión de pedidos, inventarios y perfiles de clientes con estructuras variables.
  • Redes sociales: publicaciones, comentarios, reacciones — datos de alta velocidad y estructura heterogénea.
⚠️ Cuándo NO usar MongoDB
MongoDB no es la mejor opción cuando los datos tienen relaciones muy complejas entre sí (muchos-a-muchos), cuando se requiere total consistencia transaccional (como en sistemas bancarios críticos), o cuando el esquema de los datos es perfectamente estable y uniforme.
IV · Modelo de datos

Documentos y colecciones: la unidad de almacenamiento

El documento BSON

La unidad básica de almacenamiento en MongoDB es el documento. Un documento es un conjunto de pares atributo: valor (también llamados campo: valor), escritos en formato JSON/BSON. Cada atributo es el nombre del dato; cada valor es el dato en sí.

Ejemplo de un documento de un usuario:

{ _id: 1, nombre: "Juan", edad: 30, intereses: ["música", "lectura"] }

Analicemos cada campo:

  • _id: 1 — El identificador único del documento. MongoDB lo exige en cada documento para poder distinguirlo de los demás. Si no se especifica, MongoDB lo genera automáticamente como un ObjectId (un código hexadecimal único).
  • nombre: "Juan" — Un campo de texto (String).
  • edad: 30 — Un campo numérico (Number).
  • intereses: ["música", "lectura"] — Un campo de tipo Array (lista). Un solo campo puede contener múltiples valores.
🧩 Analogía
Un documento MongoDB es como una ficha de biblioteca: tiene etiquetas (título, autor, año, ISBN…) y sus valores correspondientes. Lo novedoso es que una ficha de película puede tener "director" y "duración", mientras que la de un libro tiene "autor" y "páginas", y ambas pueden coexistir en el mismo cajón.

Las colecciones

Los documentos se agrupan en colecciones. Una colección en MongoDB es análoga a una tabla en SQL, pero sin la restricción del esquema fijo. Una base de datos MongoDB puede tener múltiples colecciones: una para usuarios, otra para productos, otra para pedidos, etc.

📌 Concepto Clave
No todos los documentos de una colección necesitan tener los mismos campos. Un documento en la colección "usuarios" puede tener el campo "teléfono" y otro no. MongoDB acepta esta heterogeneidad sin problemas.

V · Tipos de datos

Tipos de datos admitidos

Una de las grandes fortalezas de BSON frente al JSON estándar es la variedad de tipos de datos que soporta. Esto permite almacenar prácticamente cualquier tipo de información directamente en el documento, sin conversiones externas:

TipoDescripciónEjemplo
String Texto libre de caracteres "Hola mundo"
Number Número entero o decimal 30, 3.14
Boolean Verdadero o falso true, false
Date Fecha y hora ISODate("2024-01-15")
Array Lista ordenada de valores ["música", "lectura"]
Object Documento anidado (sub-documento) { calle: "Av. 5", cp: "91000" }
ObjectId Identificador único generado por MongoDB ObjectId("64a3...")
Null Ausencia de valor null
Binary Datos binarios (imágenes, archivos) BinData(...)

El tipo Object (sub-documento) es especialmente poderoso: permite anidar documentos dentro de documentos, creando estructuras jerárquicas ricas sin necesidad de tablas adicionales.

VI · Relaciones

Relaciones entre documentos

En las bases de datos relacionales, las relaciones entre entidades se implementan mediante claves foráneas y operaciones de unión (joins). En MongoDB existen dos estrategias para modelar relaciones, cada una con sus ventajas:

1. Modelado embebido (Embedding)

Consiste en incluir un documento dentro de otro como sub-documento. Es ideal cuando los datos relacionados siempre se consultan juntos.

{ _id: 1, nombre: "Ana", direccion: { calle: "Av. Xalapa 45", ciudad: "Xalapa", cp: "91000" } }
🧩 Analogía
El modelado embebido es como escribir la dirección de una persona directamente en su expediente, en lugar de guardarla en un archivero separado y poner solo un número de referencia.

2. Modelado por referencias (References)

Consiste en guardar en un documento únicamente el _id de otro documento relacionado, en lugar del documento completo. Es útil cuando los datos relacionados se consultan por separado o cuando se repiten en muchos documentos.

// Documento de pedido con referencia al usuario { _id: 101, producto: "Laptop", usuario_id: 1 // referencia al _id del usuario "Ana" }
CriterioEmbebidoPor referencias
Velocidad de lectura✅ Alta (todo en un documento)⚠️ Requiere consultas adicionales
Duplicación de datos⚠️ Posible si se repite✅ Sin duplicación
Relación adecuada para1:1, 1:N pequeñoN:M, documentos muy grandes
VII · Consultas

Consultas básicas con find()

En MongoDB, la operación para leer o buscar documentos en una colección es find(). Es el equivalente al SELECT de SQL. Su sintaxis general es:

db.nombreColeccion.find({ /* filtro */ }, { /* proyección */ })

Donde:

  • filtro es un documento que especifica las condiciones que deben cumplir los resultados.
  • proyección (opcional) indica qué campos devolver o excluir en los resultados.

Ejemplo 1: Obtener todos los documentos

db.usuarios.find({}) // Equivalente a: SELECT * FROM usuarios

Ejemplo 2: Filtrar por una condición (edad mayor a 25)

db.usuarios.find({ "edad": { "$gt": 25 } }) // Equivalente a: SELECT * FROM usuarios WHERE edad > 25

Aquí $gt es un operador de consulta que significa "greater than" (mayor que). Los operadores en MongoDB siempre van precedidos por el símbolo $.

Ejemplo 3: Proyección — seleccionar solo algunos campos

db.usuarios.find( { "edad": { "$gt": 25 } }, { "nombre": 1, "edad": 1, "_id": 0 } ) // Devuelve solo nombre y edad; excluye _id // 1 = incluir, 0 = excluir
VIII · Operadores

Operadores de consulta

MongoDB provee un conjunto rico de operadores para construir filtros de búsqueda. Se organizan en tres categorías:

Operadores de comparación

OperadorSignificadoEjemplo
$gt Mayor que (greater than) edad: { $gt: 18 }
$lt Menor que (less than) precio: { $lt: 100 }
$eq Igual a (equal) nombre: { $eq: "Ana" }
$ne Distinto de (not equal) estado: { $ne: "inactivo" }

Operadores de pertenencia

OperadorSignificadoEjemplo
$in El valor está en la lista ciudad: { $in: ["Xalapa","Veracruz"] }
$nin El valor NO está en la lista ciudad: { $nin: ["CDMX"] }

Operadores lógicos

OperadorSignificadoEjemplo
$and Ambas condiciones deben cumplirse $and: [{edad: {$gt:18}}, {activo: true}]
$or Al menos una condición se cumple $or: [{ciudad:"Xalapa"}, {ciudad:"Orizaba"}]
$not La condición NO se cumple edad: { $not: { $lt: 18 } }
🧩 Analogía
Los operadores de MongoDB son como los filtros de una tienda en línea: puedes pedir productos con precio menor a $500 ($lt), de marca X o Y ($or), y que no estén agotados ($not). Cada operador refina la búsqueda.
IX · Actualizaciones

Actualizaciones de documentos

Para modificar documentos ya existentes, MongoDB provee principalmente dos comandos:

updateOne() — Actualizar un solo documento

Modifica únicamente el primer documento que coincide con el filtro. Recibe dos argumentos: el filtro para encontrar el documento y el operador de actualización.

db.usuarios.updateOne( { nombre: "Juan" }, // filtro: busca a Juan { $set: { edad: 31 } } // actualización: cambia edad a 31 ) // Equivalente a: UPDATE usuarios SET edad=31 WHERE nombre='Juan' LIMIT 1

El operador $set indica qué campos se van a modificar sin tocar los demás campos del documento. Si el campo no existe, lo crea; si existe, lo actualiza.

📌 Concepto Clave
Es importante usar $set. Si se omite y se pasa directamente el documento de reemplazo, MongoDB reemplazará el documento completo, perdiendo todos los campos que no se especifiquen. $set hace una modificación parcial y segura.

updateMany() — Actualizar múltiples documentos

Funciona igual que updateOne(), pero modifica todos los documentos que coincidan con el filtro, no solo el primero.

db.usuarios.updateMany( { ciudad: "Xalapa" }, { $set: { region: "Centro-Norte" } } ) // Agrega el campo "region" a TODOS los usuarios de Xalapa
X · Agregaciones

Agregaciones: consultas avanzadas

El framework de agregación de MongoDB permite realizar operaciones de análisis y transformación de datos complejas, similares a las cláusulas GROUP BY, SUM, AVG, HAVING de SQL. Funciona como una cadena de procesamiento (pipeline): los documentos entran, pasan por varias etapas de transformación, y al final se obtiene el resultado.

🧩 Analogía
Imagina una línea de ensamblaje en una fábrica. Los documentos son piezas que entran por un extremo. Cada etapa de la línea hace algo: una filtra las piezas defectuosas, otra las agrupa por color, otra cuenta cuántas hay de cada color. Al final del recorrido, obtienes el resultado procesado.

Operadores de agregación principales

OperadorFunciónEquivalente SQL
$match Filtra documentos por condición WHERE
$group Agrupa documentos por un campo GROUP BY
$sum Suma los valores de un campo SUM()
$avg Calcula el promedio de un campo AVG()
$project Selecciona o transforma campos del resultado SELECT columnas

Ejemplo: Edad promedio por ciudad

db.usuarios.aggregate([ { $match: { activo: true } }, // 1. Solo usuarios activos { $group: { _id: "$ciudad", // 2. Agrupar por ciudad edadPromedio: { $avg: "$edad" } // 3. Calcular promedio de edad }}, { $project: { ciudad: "$_id", edadPromedio: 1, _id: 0 }} ])
XI · Integración con Java

Conexión a MongoDB desde Java

MongoDB provee un driver oficial para Java que permite interactuar con la base de datos desde aplicaciones escritas en ese lenguaje. Esto es especialmente relevante para proyectos Java como los que se desarrollan con Spring Boot, que es el framework que ya conoces de tu proyecto dulcería.

A continuación se analiza el código de ejemplo presentado en la diapositiva, línea por línea:

// 1. Crear la conexión al servidor de MongoDB MongoClient mongo = new MongoClient("localhost", 27017); // 2. Seleccionar la base de datos MongoDatabase db = mongo.getDatabase("miDB"); // 3. Seleccionar la colección (como elegir una tabla) MongoCollection<Document> col = db.getCollection("usuarios"); // 4. Crear un documento a insertar Document doc = new Document("nombre", "Ana").append("edad", 25); col.insertOne(doc); // Insertar el documento en la colección // 5. Buscar documentos con edad = 25 FindIterable<Document> docs = col.find(eq("edad", 25)); // 6. Iterar sobre los resultados e imprimirlos en formato JSON for (Document d : docs) System.out.println(d.toJson());

Descripción de cada paso:

  1. MongoClient: establece la conexión con el servidor de MongoDB. El parámetro "localhost" indica que MongoDB corre en la misma máquina; 27017 es el puerto predeterminado de MongoDB.
  2. getDatabase("miDB"): selecciona la base de datos con la que queremos trabajar. Si no existe, MongoDB la crea automáticamente al insertar el primer documento.
  3. getCollection("usuarios"): selecciona la colección. De igual forma, se crea si no existe.
  4. new Document(...).append(...): construye el documento Java que se insertará. Es similar a crear un mapa de clave-valor.
  5. col.find(eq(...)): ejecuta una búsqueda. El método eq() es un helper que construye la condición de igualdad.
  6. d.toJson(): convierte el documento Java a su representación textual JSON para mostrarlo en pantalla.
XII · Resumen final

Resumen final

MongoDB representa una solución práctica y madura para los problemas que las bases de datos relacionales no resuelven bien: datos heterogéneos, estructuras cambiantes, alta velocidad de escritura y lectura, y escalabilidad masiva. Aquí están los cuatro pilares a recordar:

📄

Documentos BSON

La unidad de datos es el documento: pares clave-valor en formato BSON. Flexible, anidable y sin esquema fijo obligatorio.

🗂️

Colecciones

Agrupan documentos relacionados. Equivalen a las tablas de SQL, pero sin exigir un esquema uniforme entre documentos.

🔍

Consultas y operadores

Con find() y operadores como $gt, $or, $in, se pueden filtrar documentos con gran precisión.

⚙️

Agregaciones

El pipeline de agregación ($match, $group, $avg…) permite análisis de datos complejos sin salir de MongoDB.
