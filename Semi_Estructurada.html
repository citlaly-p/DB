I · Introducción

Motivación: ¿Por qué necesitamos algo diferente?

Para entender las bases de datos semi-estructuradas, primero debemos recordar cómo funcionan las bases de datos tradicionales (las relacionales, como MySQL o PostgreSQL).

En una base de datos relacional, antes de guardar cualquier dato, se debe definir un esquema rígido: una especie de plantilla que especifica exactamente qué columnas tendrá cada tabla y qué tipo de dato irá en cada una. Por ejemplo, una tabla de empleados siempre tendrá las columnas nombre, edad y departamento, sin excepciones.

🧩 Analogía
Piensa en un formulario impreso: tiene espacios fijos para nombre, apellido y teléfono. Si alguien tiene dos teléfonos o no tiene correo electrónico, el formulario no fue diseñado para eso, y tienes un problema.

El mundo real, sin embargo, rara vez es tan ordenado. Los datos reales pueden presentar tres problemas fundamentales que los sistemas tradicionales manejan con dificultad:

  • Irregularidad e incompletez: No todos los registros tienen los mismos campos. Un restaurante puede tener número de fax; otro, no.
  • Cambio con el tiempo: La estructura de los datos puede evolucionar. Hoy un cliente solo tiene un correo; mañana la aplicación necesita guardar varios.
  • Ausencia total de esquema: Algunos datos provienen de fuentes externas (páginas web, APIs, documentos) y simplemente no tienen una estructura predefinida.

Las bases de datos semi-estructuradas nacen para resolver exactamente estos casos: datos que tienen algo de estructura, pero no la rigidez completa de un esquema fijo.

II · Modelo de datos

El modelo de datos semi-estructurado

El modelo de datos semi-estructurado se define por cuatro características esenciales que lo distinguen radicalmente del modelo relacional:

Característica Modelo Relacional Modelo Semi-Estructurado
Esquema Fijo, predefinido obligatoriamente No existe esquema fijo
Descripción de los datos Externa (en tablas y columnas) Auto-descriptivo (mediante etiquetas)
Datos incompletos Difícil de manejar (NULLs) Manejado de forma natural
Representación Tablas Grafos etiquetados (modelo OEM)

¿Qué significa "auto-descriptivo"?

En el modelo semi-estructurado, la información sobre qué es cada dato va incluida directamente junto con el dato, en forma de etiquetas. Estas etiquetas pueden cambiar dinámicamente: si aparece un campo nuevo, simplemente se agrega una nueva etiqueta; no hay que rediseñar toda la base de datos.

🧩 Analogía
Imagina una caja de cartón donde cada objeto que metes lleva una etiqueta pegada que dice qué es: "libro", "zapato", "lámpara". No necesitas que la caja tenga compartimentos fijos con nombres. Los datos se describen a sí mismos.

El modelo concreto que implementa este enfoque se llama OEM (Object Exchange Model), que veremos a continuación.

II · Modelo de datos (continuación)

OEM — Object Exchange Model

OEM es el modelo de datos formal que se usa para representar información semi-estructurada. Su idea central es representar todo como un grafo.

📌 Concepto Clave
Un grafo es una estructura matemática compuesta por nodos (puntos) y arcos (líneas que conectan esos puntos). Piensa en un mapa de ciudades: cada ciudad es un nodo y cada carretera es un arco.

En OEM, el grafo se construye con las siguientes reglas:

  • Cada entidad (cosa que queremos representar) es un nodo, llamado objeto.
  • Cada relación entre objetos es un arco que conecta dos nodos.
  • Cada arco lleva una etiqueta que describe el tipo de relación (por ejemplo: "restaurant", "name", "address").
  • Cada objeto tiene un identificador único llamado oid (object identifier).

El ejemplo de la base de datos "Guide"

La presentación usa el ejemplo de una guía de restaurantes. Imagina que tenemos un objeto raíz llamado Guide. Desde él salen tres arcos etiquetados "restaurant", cada uno apuntando a un objeto diferente (los restaurantes). A su vez, cada restaurante tiene sus propios arcos: "name", "address", "category", "nearby", etc. Cada hoja del grafo (un nodo sin arcos de salida) contiene el valor real: "Chef Chu", "El Camino Real", "92310".

🧩 Analogía
Es exactamente como un árbol genealógico, pero más flexible: los nodos son personas u objetos, y las líneas que los conectan llevan el nombre de la relación ("hijo de", "esposa de", "trabaja en"). Lo novedoso es que dos nodos pueden compartir un mismo hijo (un nodo puede tener múltiples padres), lo que lo convierte en un grafo y no solo en un árbol.
II · Modelo de datos (continuación)

Objetos OEM: atómicos y complejos

Dentro del modelo OEM, todos los nodos del grafo son objetos. Cada objeto tiene dos componentes: un identificador (oid) —un nombre único interno— y un valor. Dependiendo del tipo de valor, un objeto puede ser de dos tipos:

Tipo de objeto ¿Tiene arcos de salida? Tipo de valor Ejemplos
Atómico No (es una hoja del grafo) Un dato primitivo integer, string, image, sound, html
Complejo Sí (tiene sub-objetos) Colección de sub-objetos OEM Un restaurante, una dirección, una persona
🧩 Analogía
Piensa en un directorio de archivos en tu computadora. Una carpeta es como un objeto complejo: puede contener otras carpetas o archivos. Un archivo de texto es como un objeto atómico: contiene directamente los datos (el texto) y no se puede abrir para ver más carpetas adentro.

En el ejemplo de la guía:

  • El objeto &12 (Guide) es complejo: tiene arcos que salen hacia los restaurantes.
  • El objeto &13 con valor "Chef Chu" es atómico: es solo un nombre, no tiene nada adentro.
  • El objeto &14 (la dirección) es complejo: contiene street, city y zipcode.
III · Definición de esquemas

Esquemas y DataGuides

¿Para qué sirven los esquemas?

Dijimos que los datos semi-estructurados no requieren un esquema. Sin embargo, los esquemas siguen siendo útiles porque permiten consultar e interpretar los datos de forma eficiente. Si no sabes qué caminos existen en el grafo, ¿cómo harías una búsqueda?

📌 Concepto Clave
En el contexto de bases de datos relacionales, un esquema es la definición de las tablas, columnas y tipos de datos permitidos. Actúa como un "contrato" que restringe lo que puede almacenarse.

En las bases de datos semi-estructuradas no tenemos ese contrato rígido. La solución elegante que propone este modelo se llama DataGuide.

¿Qué es un DataGuide?

Un DataGuide es un grafo especial que se genera automáticamente a partir del estado actual de la base de datos. No está predefinido por el diseñador; se descubre examinando los datos reales. Cumple tres propiedades fundamentales:

  1. Cada camino que existe en la base de datos también existe en el DataGuide.
  2. Cada camino del DataGuide también existe en la base de datos (no incluye caminos ficticios).
  3. Cada camino en el DataGuide es único — no hay duplicados.
🧩 Analogía
Imagina que tienes miles de documentos en una oficina, cada uno con campos distintos. Un DataGuide sería como el índice general que dice: "en esta colección de documentos, existen los campos: nombre, dirección, teléfono, fax, correo…" — pero sin repetir ninguno, aunque aparezca miles de veces en los documentos originales. Es un resumen de la estructura, no de los datos.

Ejemplo visual

Si la base de datos tiene cuatro personas: una con nombre y teléfono; otra con nombre y fax; otra con nombre, teléfono y fax; y otra solo con nombre — el DataGuide tendrá un único nodo "persona" con tres posibles hijos: nombre, teléfono y fax. Condensa toda la variedad estructural en una sola representación compacta.

En el ejemplo del restaurante, la base de datos tiene dos restaurantes con campos diferentes: uno tiene Owner y Phone; el otro tiene Manager. El DataGuide los combina en un único nodo "Restaurant" que muestra todos los posibles sub-campos: Name, Entree, Phone, Owner y Manager.

IV · Lenguaje de interrogación

Lenguaje de interrogación: LOREL

LOREL (Lightweight Object REpository Language) es el lenguaje de consulta diseñado específicamente para trabajar con datos OEM. Su propósito es permitir al usuario hacer preguntas sobre el grafo de datos de forma cómoda y declarativa.

  • Es un lenguaje declarativo: el programador dice qué quiere obtener, no cómo buscarlo.
  • Es conceptualmente similar a SQL, pero adaptado para grafos en lugar de tablas.
  • Se considera una extensión de OQL (Object Query Language).
  • La interrogación se basa en expresiones de camino (path expressions): secuencias de etiquetas que describen cómo navegar el grafo para llegar al dato deseado.

Sintaxis básica

SELECT <lista de expresiones de camino> [FROM <variable>] [WHERE <condición>]

Expresiones de camino (path expressions)

Una expresión de camino es simplemente la ruta que se recorre en el grafo para llegar a un dato, usando puntos (.) para separar las etiquetas:

Guide.restaurant.address.zipcode

Esto significa: "parte del nodo Guide, ve al arco 'restaurant', luego al arco 'address', y finalmente al arco 'zipcode'". Si hay varios restaurantes, LOREL los recorre todos automáticamente.

Ejemplo 1: Consulta simple

Recuperar el objeto completo Guide:

SELECT Guide

Ejemplo 2: Filtrar con WHERE

Encontrar las direcciones de los restaurantes cuyo código postal sea 92310:

SELECT X.address FROM Guide.restaurant X WHERE X.address.zipcode = 92310 -- Resultado: address street "El Camino Real" city "Palo Alto" zipcode 92310

Nótese que la variable X itera sobre todos los restaurantes del grafo. El WHERE filtra solo aquellos que cumplen la condición.

Ejemplo 3: Misma consulta sin variable

LOREL también permite escribir la misma consulta sin usar una variable intermedia, usando el camino completo directamente:

SELECT Guide.restaurant.address WHERE Guide.restaurant.address.zipcode = 92310

Ejemplo 4: Operadores especiales para datos irregulares

Este ejemplo busca nombres y código postal de restaurantes "baratos", manejando el hecho de que la dirección puede o no existir:

SELECT Guide.restaurant.name, Guide.restaurant(.address)?.zipcode WHERE Guide.restaurant.% grep "cheap"

LOREL incluye operadores especiales para manejar la irregularidad de los datos:

  • ? (signo de interrogación): indica que ese segmento del camino es opcional. Si el restaurante no tiene dirección, en lugar de generar un error, simplemente retorna vacío.
  • % (porcentaje): es un comodín que coincide con cualquier sub-objeto del nodo actual. Es como decir "cualquier campo que tenga este restaurante".
  • grep: función que devuelve verdadero si la cadena de texto buscada aparece en el valor del objeto.
  • like: funciona igual que en SQL, para hacer comparaciones con patrones (%252 significa "termina en 252").
  • | (barra vertical): representa alternativa, como un "o" lógico en el camino (.Room%|.Cubicle significa "busca en Room o en Cubicle").

Ejemplo 5: Consulta sobre DBGroup

Oficinas de los miembros cuya edad supere los 30 años:

SELECT DBGroup.Member.Office WHERE DBGroup.Member.Age > 30 -- Resultado (heterogéneo): Office "Gates 252" Office Building "CIS" Room "411"

Observa el resultado: el primer Office es atómico (un string directo) y el segundo es complejo (tiene Building y Room). LOREL devuelve ambos tal como son, sin forzar una estructura uniforme. Esto refleja perfectamente la naturaleza semi-estructurada de los datos.

Ejemplo 6: Uso combinado de operadores

Nombres de miembros cuyo número de oficina termine en 252:

SELECT DBGroup.Member.Name WHERE DBGroup.Member.Office(.Room%|.Cubicle)? like "%252" -- Resultado: Name "Jones" Name "Smith"

Esta consulta busca en la oficina de cada miembro: si tiene subobjeto Room (con cualquier sufijo) o Cubicle (o ninguno de los dos, gracias al ?), y verifica si ese valor termina en "252".

V · Resumen

Resumen final

Las bases de datos semi-estructuradas representan una evolución del modelo relacional, diseñada para lidiar con la realidad de que muchos datos del mundo no encajan en tablas rígidas. Aquí están los tres puntos fundamentales que debes retener:

🗂️

Flexibilidad estructural

Permite representar información sin una estructura regular y fija. Los datos pueden ser incompletos, irregulares o cambiar con el tiempo sin rediseñar la base de datos.

🔍

Esquema dinámico

La noción de esquema sigue siendo importante para poder consultar los datos, pero no tiene que estar predefinido. Los DataGuides lo generan automáticamente desde los datos reales.

🧭

Navegación por grafos

Los lenguajes de interrogación como LOREL proveen mecanismos poderosos de navegación mediante expresiones de camino, comodines y operadores para manejar opcionaliedad e irregularidad.
